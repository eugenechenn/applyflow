# ApplyFlow 项目重构总结

日期：2026-04-14

## 1. 当前 Hiring Decision OS 项目复盘

### 项目本质
`Hiring Decision OS` 更像一个产品化方向明确、结构设计不错的 AI 决策分析 Demo / MVP 方案，但还不是被证实跑通的强 Agent 系统项目。

### 当前项目做了什么
- 输入 `JD + 简历 + 面试 transcript`
- 输出结构化复盘、Rubric 评分、风险、缺失信号、下一轮问题和推荐结论
- 定义了多阶段分析链路：
  - `Transcript Analyzer`
  - `Role Matcher`
  - `Evidence Checker`
  - `Decision Synthesizer`
- 前端为静态页面触发 `analyzeCandidate(payload)`
- 后端提供 `/analyze_candidate`
- 优先调用 GLM，失败时回退到 heuristic fallback

### 架构判断
- 架构类型：更像 `pipeline + prompt orchestration`
- 不是严格意义上的强 multi-agent 协作系统
- 现有材料看不到：
  - agent 间状态共享
  - 冲突仲裁
  - 长任务状态管理
  - 失败恢复
  - 运行审计与评估闭环

### 输入输出设计
输入：
- 岗位 JD
- 候选人简历
- 面试记录 transcript
- rubric version

输出：
- overall recommendation
- confidence
- summary
- rubric scores
- strengths / risks / missing signals
- next round questions
- evidence quality check

### 当前项目真实能力
真实解决的问题：
- 让招聘决策更结构化
- 让判断更可解释
- 减少只靠直觉的面试复盘

真实能力上限：
- 偏分析和建议
- 不能真正推动招聘流程闭环执行

### 解决了什么问题
解决的是：
- 招聘判断分散
- 评价标准不统一
- 面试结论证据不足
- 下一轮问题设计缺少结构

### 成熟度判断
- 不是纯玩具
- 但也不是 production
- 更接近：`分析型 Demo / MVP`

### 哪些部分是表面复杂
- `Multi-Agent` 命名可能大于实际系统深度
- `Decision OS` 命名偏大，容易让面试官期待完整执行闭环
- 如果没有真实评估数据，容易显得是 prompt 拆分

### 哪些部分真正有价值
- 从“总结工具”提升到“证据化决策支持”是对的
- 输出 schema 完整，有业务味道
- 人机边界比较清楚，不直接替代最终录用决策
- 有基础系统化意识，不只是聊天壳

## 2. 面试竞争力评估

### 优势
- 选题成立，招聘决策本身是高价值场景
- 输入输出完整，不是单轮聊天工具
- 有结构化分层意识
- AI PM 叙事较强，强调证据链和人机边界

### 关键不足
- 缺少真实用户验证
- 没有行动闭环：`input -> analysis` 多，`decision -> action -> feedback` 少
- agent 更像 prompt 拆分，不像真实协作系统
- 缺少工程硬度证据：
  - benchmark
  - 链路追踪
  - bad case
  - 审计日志
  - 评估数据
- 缺少执行能力，更多是判断，不是行动
- 仓库没有足够实现细节来证明工程落地

### 严格结论
如果用于面试：
- 面 AI PM：可讲，但要谨慎控制预期
- 面 AI Agent / AI Engineering：会被深挖出闭环不足和工程深度不足

### 面试竞争力评分
- 综合评分：`68/100`
- 如果讲得稳：可到 `72/100`
- 如果被深挖：可能掉到 `60+`

## 3. 新主项目方向结论

不建议继续把 `Hiring Decision OS` 作为主项目。

建议改做：

# ApplyFlow / Job Search Copilot

一句话定义：

> 一个半自动求职执行 Agent，帮助求职者从“看到岗位”走到“完成高质量申请与后续跟进”，而不是停留在信息收集和简历建议层。

## 4. 为什么新项目更适合当前求职目标

结合当前个人情况：
- MBA / 偏业务背景
- 暂无特别明确的目标岗位
- 更想打动 PM 面试官和业务面
- 可以投入 3 周时间高强度完成 MVP
- 有长期真实使用条件
- 核心痛点是：
  - 海投低效
  - 改简历慢
  - 跟进混乱
  - 复盘无体系
  - 噪音太多

因此，`ApplyFlow` 比 `Hiring Decision OS` 更优：
- 用户价值更直接
- 你自己就是高频真实用户
- 更容易形成闭环
- 更适合 PM / 业务叙事
- 更容易在 3 周内做出可展示 MVP

## 5. 新项目核心定位

项目名建议：
- `ApplyFlow`
- 或 `Job Search Copilot`

更推荐：`ApplyFlow`

推荐定位表述：

> ApplyFlow 是一个半自动求职执行 Agent，帮助求职者完成从筛岗、定制简历、投递准备、状态跟进到面试复盘的一整套求职执行闭环。

## 6. 产品边界

### 系统负责
- 收集岗位信息
- 清洗岗位描述
- 做匹配评估
- 判断值不值得投
- 生成定制版简历内容
- 生成自我介绍 / 问答草稿 / 投递备注
- 管理申请状态
- 生成跟进提醒
- 面试后生成复盘建议

### 用户负责
- 决定是否投递
- 最终点击提交
- 输入实际面试反馈和结果

### 为什么不做全自动投递
- 多平台适配成本高
- 登录和验证复杂
- 正式投递属于高风险动作
- 3 周内不值得把资源投入在自动化适配细节上

结论：
- 当前最优边界是 `半自动闭环`
- 不是全自动
- 也不是纯建议工具

## 7. 新项目是否适合做成 Agent 项目

适合，原因不是“能拆多个 prompt”，而是：
- 任务天然是多步骤
- 需要状态管理
- 需要跨系统执行
- 需要反馈迭代
- 需要明确人机边界

## 8. 推荐的系统结构

建议采用：`单主控 + 4/5 个功能 Agent`

### 1. Job Ingestion Agent
职责：
- 输入岗位链接或文本
- 结构化提取岗位信息
- 去重和清洗

### 2. Fit Evaluation Agent
职责：
- 结合用户背景和偏好判断匹配度
- 输出匹配分、风险、缺口、建议

### 3. Application Prep Agent
职责：
- 生成岗位定制版内容：
  - 简历 bullet 改写
  - 项目表述
  - 自我介绍
  - 常见申请问答草稿

### 4. Pipeline Manager Agent
职责：
- 管理申请流程状态：
  - 待评估
  - 待投递
  - 已投递
  - 待跟进
  - 面试中
  - 已拒绝
  - Offer
- 生成待办和跟进建议

### 5. Interview Reflection Agent
职责：
- 面试后复盘
- 识别问题和改进点
- 反推求职策略调整

### 主控层职责
- 编排 agent 调用
- 维持状态和上下文
- 避免各 agent 各说各话

## 9. 项目核心闭环

完整闭环应该是：

`岗位输入 -> 匹配决策 -> 定制申请材料 -> 人工确认并投递 -> 状态跟踪 -> 面试复盘 -> 反馈回到策略`

这条链路是整个项目的灵魂。

## 10. 三周 MVP 计划

### 第 1 周：跑通最小闭环
目标：
让用户输入岗位后，系统输出“值不值得投 + 为什么 + 简历怎么改 + 记录到看板”。

必须完成：
- 岗位输入页
- 用户画像页
- 匹配评估
- 简历定制生成
- 岗位状态记录
- 简单看板

### 第 2 周：加强执行感和管理感
目标：
从分析器升级为求职工作台。

新增：
- 投递前 checklist
- 常见问答草稿
- 跟进提醒
- 状态流转
- 每日待办清单

### 第 3 周：补反馈闭环和展示层
目标：
更像完整产品，而不是脚本集合。

新增：
- 面试复盘页
- 转化统计
- 策略反馈
- 2 到 3 个真实 case
- 完整项目说明文档

如果还有余力：
- 自动生成投递邮件草稿
- 同步待办到 Notion / 飞书表格 / 本地任务列表

## 11. 先不要做的内容

以下功能短期不要碰：
- 全平台自动登录和自动投递
- 复杂浏览器自动化
- 大规模岗位抓取适配
- 复杂长期记忆系统
- 重型 RAG
- 复杂自学习模型训练

原因：
- 容易失控
- 工期不划算
- 容易重回“看起来高级，但不落地”的老路

## 12. 推荐的个人标签

建议打造的人设标签：

> 把 AI 从分析工具做成执行工作流的 AI Native PM 候选人

可拆成三点：
- 业务问题抓得准
- 系统边界清楚
- 能把 AI 做进流程，而不是只做聊天窗口

## 13. 面试时最该强调的 PM 价值点

### 1. 场景为什么成立
求职者最大问题不是信息少，而是执行链路断裂。

### 2. 为什么不是普通 workflow
岗位匹配、简历定制、复盘优化存在非确定性，需要 LLM 做语义理解和策略生成。

### 3. 为什么不是纯聊天产品
聊天只能给建议，不能管理求职状态和推进任务。

### 4. 为什么不是全自动
正式投递属于高风险动作，当前最优边界是“半自动 + 人确认”。

### 5. 如何验证价值
用业务指标而不是“感觉变聪明”：
- 单岗位准备时间是否下降
- 有效投递数是否提升
- 申请状态是否更清晰
- 面试复盘是否更系统
- 面试转化率是否改善

## 14. 风险边界建议

当前版本建议坚持：
- 系统不能自动提交正式申请
- 系统可以生成草稿和准备材料
- 所有对外发送动作都要用户确认
- 系统不能编造经历、项目、数据
- 用户画像和结论必须可编辑
- 面试复盘只做建议，不做绝对判断

这样做的好处：
- 工程复杂度可控
- 伦理和合规风险更低
- 面试时边界感更强

## 15. 新项目面试竞争力预估

如果按这个方向认真做，并连续使用 2 到 3 周：
- 预计竞争力：`78/100`

如果再补上：
- 真实使用数据
- 2 到 3 个 bad case
- 清晰状态机
- 评估指标面板

则有机会到：
- `82-85/100`

## 16. 与 Hiring OS 的关系建议

建议保留 `Hiring Decision OS`，但降级为副项目。

推荐叙事：

> 我最初从招聘方视角做了一个 Hiring Decision OS，但后来发现它停留在分析层，缺少真实行动闭环。于是我进一步从求职者侧重构了一个半自动求职执行 Agent，把 AI 从“给建议”推进到“推动任务完成”。

这会体现出：
- 你有复盘能力
- 你能主动识别项目问题
- 你不是死抱旧项目

## 17. 后续正式开始时要做的四件事

下一轮建议直接进入：
- 项目 PRD v1
- MVP 信息架构和页面结构
- Agent 架构图和状态机
- 面试讲稿与包装文案

## 18. 当前最终结论

主项目方向明确建议：

# ApplyFlow

定位：
- 半自动求职执行 Agent
- 面向真实求职者
- 强调闭环、状态管理、执行辅助和反馈优化

战略取舍：
- 放弃全自动投递幻想
- 优先做窄而完整的闭环
- 用真实使用和业务指标证明价值

目标：
- 用 3 周做出一个对 PM / 业务面有说服力、并且具备 Agent 味道的真实项目
