# AGENTS.md

## 目标
本文件用于说明当前项目的本地运行、文件处理、代码规范和文档维护规则。  
开始当前项目任务时，优先读取本文件，并按这里的规则执行。

---

## 文件与编码
- 所有文件读写默认使用 UTF-8 编码
- 修改文件时不要改变原有编码
- 处理含中文内容时，不使用 sed/awk，优先使用 Python 或 Node.js
- PowerShell 读取中文文件时：
  - 先执行 `chcp 65001`
  - 读取文件使用 `Get-Content -Encoding UTF8`
- 代码注释统一使用中文

---

## Python 环境
- 优先使用项目本地虚拟环境
- Windows：`.venv/Scripts/python.exe`
- Unix：`.venv/bin/python`
- 不直接使用系统 `python`
- 如果 `.venv` 不存在，先创建再运行：
  - `python -m venv .venv`

---

## 文档读取顺序
按需读取，不强制全读：

1. `CONTEXT.md`：看当前目标、当前进度、下一步和注意事项
2. `DECISIONS.md`：看为什么这样设计
3. `TIMELINE.md`：看最近在哪些日期改过什么
4. `VALIDATION_SPEED_POLICY.md`：看验证分层、提速策略、磁盘/DB 安全策略
5. `README.md`：看项目简介、功能和基本使用方式
6. `ARCHITECTURE.md`：看模块职责、调用关系和结构设计
7. `DEPLOYMENT.md`：看本地运行、测试、Git 和部署流程
8. `RESEARCH.md`：看外部参考结论，避免重复搜索

简单任务默认只读 `CONTEXT.md`。  
遇到具体问题时，再按需要补充读取其他文档。

## 文档维护规则

### CONTEXT.md
始终保持极简，只保留当前工作现场，内容包括：
- 当前目标
- 当前进度
- 下一步
- 注意事项
- 最后更新时间

规则：
- 内容过时直接覆盖
- 不保留历史版本
- 保持简短，便于新线程快速接手

### TIMELINE.md
用于按日期索引项目改动，便于快速定位问题来源。

规则：
- 每次阶段性变更后追加一条
- 每条一行，一句话即可
- 只写“做了什么”，不展开原因和过程

示例：
- 2026-04-17：调整菜单结构字段，前端改为分组展示
- 2026-04-15：从 OCR-first 改为视觉模型直出

### DECISIONS.md
用于记录关键设计决定。

适用场景：
- 技术路线改变
- 方案取舍
- 明确放弃某方案及原因

规则：
- 只写“为什么这样做”
- 不写长过程
- 一条决策尽量独立成段，便于后续查找

### README.md
用于面向使用者说明项目。

仅在以下情况更新：
- 功能有变化
- 项目说明需要补充
- 运行方式有明显变化

内容保持简洁，重点包括：
- 项目是做什么的
- 技术怎么组成
- 功能列表
- 待办事项

### ARCHITECTURE.md
用于记录项目结构。

仅在结构变化时更新，内容包括：
- 模块职责
- 调用关系
- 核心设计

### DEPLOYMENT.md
用于记录操作流程。

仅在以下内容变化时更新：
- 本地运行方式
- 测试方式
- Git 流程
- 部署流程

内容保持简短清晰，重点包括：
- 本地启动与测试方法
- 常用测试命令
- Git 基本流程（提交 / 分支 / 合并）
- 部署步骤
- 部署后验证方法
- 必要注意事项

### RESEARCH.md
用于记录外部参考结论。

规则：
- 仅在进行了 skills.sh、GitHub 或其他外部搜索后更新
- 记录结论，不记录冗长搜索过程
- 目标是避免重复搜索

## 代码习惯
- 调试问题时，先确认根因和影响范围，再修改，不做表面修复
- 涉及数据删除、数据库变更或高风险操作，必须先确认
- 需求不清晰时，先提问再实现，不自行扩大需求
- 避免重复执行任务或无意义多轮测试

---

## 代码规范
- 单个文件尽量控制在 500 行以内，过长时考虑拆分
- 函数尽量保持单一职责
- 文件顶部写简短中文注释，说明用途
- 关键函数写一行中文注释说明作用
- 不硬编码配置、路径或密钥，统一放入配置文件
- 所有错误必须有明确提示或日志，不能静默失败

---

## 外部参考
- 只有在当前项目确实需要外部方案参考时，才搜索 skills.sh 或 GitHub
- 优先先看 `RESEARCH.md`，若已有结论则不重复搜索
- 搜索后将结论写入 `RESEARCH.md`

## Review 规范（新增）

### 子线程 review 默认策略（必须遵守）
1. 先做 `30-60s` 的只读 smoke test，再进入 `900s` 正式 review。  
   smoke test 需验证目录、模型权限、沙箱、输出文件和回包链路可用。
2. review 必须分阶段进行：  
   - 第一阶段仅 review 本轮改动文件与直接相关回归脚本  
   - 第二阶段按需扩展到下一批新增改动  
   禁止一次性把整条大任务、整页大文件、整仓上下文全部丢给子线程。
3. 子线程未返回前，主线程继续推进不冲突工作（实现、验证、取证、埋点或下一阶段收敛），不得空等。
4. review 范围按风险分轨：  
   - 模块/页面层改动：默认聚焦本轮改动  
   - 共享层改动：必须扩展到“受影响功能”，不能只看 diff
5. 共享层 review 任务必须显式写出“影响面”：  
   - 改动的是哪一层共享能力  
   - 可能影响哪些功能  
   - 本轮要求重点核查哪些相邻流程  
   不能只写“帮我 review 这个文件”。
6. 只有拿到真实 review 返回内容，才能声明“本轮 review 已完成”。  
   超时时只能标注“仍在等待/本轮未回包”，不能写“已 review 通过”。
7. timeout 降级自动触发：  
   出现超时、卡在仓库快照/状态检查、或同类命令连续无响应时，必须自动进入降级流程，不等待额外提醒。  
   “同类命令连续无响应”默认定义为：连续 `2` 次同类 review/prototype 命令未在各自预算内拿到有效正文返回。  
   “有效正文返回”必须包含实际分析、patch 或 findings；仅启动信息、空结果、路径/目录错误不算有效返回。

### Review 安全边界门禁（必须遵守）
1. review 不能只等同于“代码正确性检查”，必须同时覆盖：  
   - 正确性问题（需求满足、显式 bug、异常路径、漏判分支、数据丢失、测试缺口）  
   - 安全边界问题（fast path 绕过校验、source-of-truth 绕过、缓存失效风险、共享层扩散误伤、主路径更快但门禁被绕过）
2. 若安全边界问题未被明确检查，不得声明“review 已完成/已通过/无风险”。
3. 若发现安全边界问题、共享层误伤风险、source-of-truth 绕过或缓存失效风险，必须先止损：  
   暂停合并、提交、发布、同步到运行目录或“已完成”声明，直到风险被修复、回退，或由用户明确接受残余风险并记录到 issue 台账。
4. 解封前必须重做同范围或更大范围 review，并在台账记录 `Disposition`：  
   `Blocked/Unblocked` 状态、owner、解封条件、复审 `SESSION_ID` 或等效证据。  
   未完成复审闭环，不得将“已处理”视为“可继续”。

### Review 提示词强制项（必须遵守）
派发 review 时禁止使用“帮我看下有没有 bug”这类弱提示词。  
提示词必须显式要求对抗性场景检查，至少覆盖：
- 边界输入对抗：空值、空数组/空 Hash、极值、乱序、重复项、缺字段、旧字段别名
- 状态错位对抗：request payload 与 DB 不一致、快照与最新模型不一致、缓存与文件/云端资产不一致、运行目录与源码不一致
- 共享层扩散对抗：同一 helper 被多入口复用时，非目标入口是否误入新路径
- 缓存失效对抗：缓存命中/未命中、先缺失后出现、同路径内容更新、不同模型/不同用户复用同缓存
- 回归链路对抗：当前修复路径通过时，相邻功能、历史修复点、冷启动流程是否仍覆盖

review 任务必须明确写出：影响面、禁止改动层、预期不变行为、重点误伤路径。  
主线程自审也必须执行同组对抗场景；即使未派发子线程，也要在最终说明或 issue 台账记录已检查项。

### Review 结果最低结构（必须遵守）
有效 review 返回至少回答以下六项，缺任一项仅算“部分 review”，不得算通过：
1. `Correctness Findings`：代码正确性发现
2. `Boundary/Safety Findings`：安全边界、误入路径、source-of-truth、缓存失效发现
3. `Adversarial Scenarios Checked`：实际检查过的对抗场景
4. `Blast Radius & Adjacent Regression Assessment`：影响面与相邻回归评估
5. `Residual Risks`：仍未验证或需冷启动/实测确认的风险
6. `Disposition`：本轮是否 `Blocked`、解封条件、owner、若曾阻塞则对应复审闭环证据

## 执行优先级（最高优先级，必须遵守）

当 AGENTS.md 被读取后：

1. AGENTS.md 中的流程规则高于默认执行习惯
2. 若用户提示词未明确豁免 review，则 Review 规范必须自动执行
3. 任一阶段任务完成后：

   * 必须自动执行 Focused Review
   * 未完成 review 不得声明完成
4. CONTEXT/TIMELINE 更新后，
   仍需执行 Review 闭环
5. 若提示词与 AGENTS.md 流程冲突：
   默认优先遵守 AGENTS.md，除非用户明确要求覆盖
6. 若用户未明确要求全量重跑，验证执行默认遵守 `VALIDATION_SPEED_POLICY.md` 的分层策略（L1/L2/L3）
7. 涉及磁盘占用、DB 写入、批量临时文件生成时，必须先按 `VALIDATION_SPEED_POLICY.md` 的安全规则约束执行


若子线程返回“无发现”但未说明安全边界与对抗场景，主线程必须追问或重发 review，不得直接接受。  
以上各项必须尽量证据化，至少包含文件/函数/调用链、触发条件、已执行命令/测试/冷启动结果或未覆盖原因。  
只有标题、空泛判断或“看起来没问题”不算有效 review 证据。
