## 🧠内容总结

这组截图深刻指出了在 AI PM 面试中拉开差距的核心要素：不要只停留在“我用 GPT 做了个功能”，而要展现真正的 **AI Native（AI原生）思维**。这种思维体现在三个维度：用**设计哲学**定义产品边界（敢于放弃），用**AI原生工作流**重组团队协作模式（AI为第一生产力），以及具备精准的**技术判断力**（清晰界定 Rule-Based 与 LLM 的边界及成本意识）。

## 📌关键信息

- **理念先行**：产品功能背后是一套信念（Design Philosophy）。优秀的 AI PM 能清晰表达产品“选择解决什么”以及“主动放弃什么”。
    
- **人机角色的范式反转**：在 AI Native 组织中，AI 变成了“第一生产力/核心执行者”，而人类的角色转变为：提供 Context（上下文）、解决卡点（Unblock）以及对最终结果负责。
    
- **绩效衡量标准改变**：工作流的改变意味着绩效不再按 PR（代码提交）数量或工作时长衡量，而是按“人与 AI 协作的产出效果”评估。
    
- **拒绝“拿着锤子找钉子”**：必须明确 LLM 的精确加成点在哪。不要什么都扔给最贵的模型，要能清晰拆解产品步骤，找出必须花 Token 让大模型做决策的动态复杂环节。
    

## 🧩方法/模型（可复用）

**1. AI 产品“设计哲学”表达框架**

在路演或面试介绍产品时，遵循以下三段论结构：

- **① 一句话核心理念**：高度浓缩你的产品价值观（例如：“找工作不应是海投，而是让信息主动找你”）。
    
- **② 基于哲学做重（Focus）**：为了实现上述理念，你选择优先解决的痛点及采取的核心方案。
    
- **③ 主动放弃的边界（Trade-off）**：体现产品勇气的断舍离（例如：砍掉大而全的搜索，只留精准信号匹配）。
    

**2. AI Native 团队协作模型**

重构传统的“PM -> UI -> 研发 -> 测试”流水线，转变为按产出分工：

- **Product Owner (PO)**：负责定义需求、最终交付成果以及“做到什么程度”（不写代码，定方向）。
    
- **AI Architect / Builder**：核心执行者，负责用 AI 快速构建产品，管理 AI 的输出质量和 Context。
    
    _(核心复用点：规划个人或小型创业团队工作流时，可直接套用此模式，实现 3-5人 达到过去 5-10人 的产能)_
    

**3. LLM 价值判别器（Why LLM）**

用于梳理业务逻辑，决定技术选型的切分模型：

- **分界线判断**：确定性逻辑 $\rightarrow$ 使用 `Rule-Based`（传统规则/搜索/自动化）；动态复杂需决策 $\rightarrow$ 使用 `LLM Intelligence`。
    
- **LLM 三大核心能力对号入座**：
    
    - **推理 (Reasoning)**：用于动态、模糊场景下的判断与决策。
        
    - **领域知识 (Domain Knowledge)**：用于跨领域的理解与知识点串联。
        
    - **生成能力 (Generation)**：用于产出人类可读的内容或代码。
        
- **ROI 选型策略**：简单任务用小模型，复杂任务用大模型，建立清晰的 Token 成本意识。
    

## 🎯适用场景

- **AI 产品经理求职面试**：用于组织作品集宣讲的逻辑主线，降维打击只会背诵 Prompt 的竞争者。
    
- **AI 产品立项/BP撰写**：在向老板或投资人汇报时，用“设计哲学”定调，用“Why LLM”自证技术合理性。
    
- **个人超级个体/草台班子组建**：参考 AI Native Workflow，优化个人的 Agent 协作外包流程。
